## 📌 들어가며

입력 함수를 처음 쓰다 보면 "왜 두 번째 입력을 받지도 않고 프로그램이 끝나버리지?" 하는 순간을 만난다. 범인은 대부분 **버퍼(buffer)**다. 이번 글에서는 버퍼가 무엇인지, 그리고 엔터키(`\n`)가 어떻게 입력을 꼬이게 만드는지 살펴본다.

> **버퍼(buffer)란?**
> 어떤 장치에서 다른 장치로 데이터를 보낼 때 생기는 **시간 차이나 속도 차이를 조정**하기 위해, 데이터를 일시적으로 기억해두는 장치. 순화어로는 **'완충기'**라고도 한다.

---

## 1. 버퍼는 "중간 보관소"다

우리가 키보드로 입력한 데이터는 곧장 프로그램으로 가는 게 아니라, 중간의 버퍼에 잠시 머물렀다가 처리된다.

![Desktop View](/assets/img/Programming-Language/C/Buffer/1.png){: width="100%" }

위 이미지의 버퍼에는 `\n`(줄바꿈, 엔터키)이 들어가 있다. 이렇게 버퍼는 **우리가 입력한 데이터를 일시적으로 보관하고 처리하는 중간 지점**이라고 보면 된다.

```
키보드 입력  ──►  [ 버퍼 (중간 보관소) ]  ──►  프로그램(scanf)
                   a  \n  ...
```

---

## 2. 문제 상황 — 두 번째 입력이 씹힌다

문자 하나를 입력하면 그 문자가 출력되고, 또 다른 문자를 입력하면 그것도 출력되도록 하고 싶었다.

![Desktop View](/assets/img/Programming-Language/C/Buffer/2.png){: width="100%" }

그런데 `a`를 입력하고 엔터를 눌렀더니?

![Desktop View](/assets/img/Programming-Language/C/Buffer/3.png){: width="100%" }

두 번째 문자를 입력할 새도 없이 프로그램이 끝나버린다. **왜일까?**

---

## 3. 원인은 버퍼에 남은 `\n`

답은 버퍼에 있다. `a`를 입력할 때 우리는 `a`를 누르고 **엔터키**도 눌렀다. 그래서 버퍼에는 `a`와 `\n`, **두 개의 문자**가 들어간다.

> ⚠️ **핵심**: `%c`는 버퍼에서 **무조건 한 개의 문자**를 가져간다. 그리고 그 대상에는 `\n`(엔터)도 **포함**된다.

순서대로 보면 이렇다.

```
1) 사용자가  a  Enter  입력   →  버퍼:  [ a ][ \n ]
2) 첫 scanf가 %c로 한 글자    →  a 를 가져감 → printf가 a 출력
                                버퍼에 [ \n ] 이 남음!
3) 두 번째 scanf가 %c로 한 글자
   → 버퍼에 이미 \n 이 있으므로
     사용자 입력을 기다리지 않고 \n 을 그대로 가져감
   → 화면에는 공백처럼 출력되고 끝
```

즉, 두 번째 입력이 "씹힌" 게 아니라, **이미 버퍼에 남아 있던 엔터(`\n`)가 두 번째 입력 자리를 차지해버린** 것이다. 그래서 두 번째 출력이 공백으로 표시된다.

> 💡 이런 문제는 보통 `%c` 앞에 공백(`scanf(" %c", ...)`)을 넣거나, 버퍼를 비워주는 방식으로 해결한다. 원리를 알면 대처법도 자연스럽게 이해된다.

## 📝 정리

| 개념 | 한 줄 정의 |
|------|------|
| **버퍼** | 입력/출력 데이터를 일시적으로 보관하는 중간 저장소 |
| **`\n`의 함정** | 엔터키도 하나의 문자로 버퍼에 남는다 |
| **`%c`의 특성** | 버퍼에서 무조건 한 문자를 가져감 (`\n` 포함) |

버퍼의 동작 원리는 앞으로 입출력을 다룰 때 정말 중요한 요소다. 입력이 이상하게 동작할 때 "혹시 버퍼에 `\n`이 남아 있나?"를 먼저 떠올릴 수 있게 되면, 디버깅이 훨씬 수월해진다.
