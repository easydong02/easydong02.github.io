## 📌 들어가며

DB 접속 정보(URL, 계정, 비밀번호)를 소스 코드에 하드코딩하면, 개발·운영 환경이 바뀔 때마다 코드를 고쳐야 한다. 이런 **환경별로 달라지는 리소스**를 코드 밖에서 관리하도록 도와주는 것이 **JNDI**다.

> **JNDI(Java Naming and Directory Interface)란?**
> 자바 애플리케이션에서 **네이밍·디렉터리 서비스에 접근**하기 위한 API. 이름(name)에 객체(object)를 매핑해두고, 애플리케이션은 그 **이름만으로 리소스에 접근**한다.

JNDI는 주로 다음에 사용된다.

| 용도 | 예 |
|------|------|
| 데이터베이스 연결 정보 | DataSource |
| 메일 서버 연결 정보 | Mail Session |
| 리소스 설정 정보 | 파일 경로, 디렉터리 등 |

```
애플리케이션 ──"jdbc/myDataSource" 이름으로 lookup──► JNDI ──► 실제 DataSource
                                                    (이름 ↔ 객체 매핑)
```

---

## 1. 서버 환경 설정 (server.xml)

JNDI를 쓰려면 먼저 서버에서 리소스를 등록해야 한다. 톰캣(Tomcat)의 경우 **`server.xml`**(또는 `context.xml`)에서 설정한다.

```xml
<Context>
    <!-- JNDI 설정 -->
    <Resource name="jdbc/myDataSource"
              auth="Container"
              type="javax.sql.DataSource"
              driverClassName="com.mysql.jdbc.Driver"
              url="jdbc:mysql://localhost:3306/mydb"
              username="user"
              password="password"
              maxTotal="100"
              maxIdle="30"
              validationQuery="SELECT 1"
              maxWaitMillis="10000"/>
</Context>
```

> ⚠️ **실무에서 삽질한 포인트**: `validationQuery`를 작성했다면 **반드시 그 DB에 맞는 쿼리**를 써야 한다. 예를 들어 Oracle이라면 `SELECT 1`이 아니라 **`SELECT 1 FROM DUAL`**을 써야 한다. 이걸 빼먹어서 회사에서 한참 헤맸다.

## 2. 애플리케이션에서 JNDI 사용

등록된 JNDI 리소스는 **`InitialContext`**로 조회(lookup)한다.

```java
import javax.naming.Context;
import javax.naming.InitialContext;
import javax.sql.DataSource;
import java.sql.Connection;

public class JndiExample {
    public static void main(String[] args) {
        try {
            Context context = new InitialContext();
            // 이름으로 DataSource를 조회
            DataSource dataSource =
                (DataSource) context.lookup("java:comp/env/jdbc/myDataSource");
            Connection connection = dataSource.getConnection();
            // Connection으로 DB 작업 수행
            connection.close();   // 작업 완료 후 연결 닫기
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
```

`context.lookup("java:comp/env/jdbc/myDataSource")` — server.xml에 등록한 **이름**으로 실제 DataSource를 받아온다. 코드에는 접속 정보가 전혀 노출되지 않는다.

---

## 3. JNDI 설정의 장점

| 장점 | 설명 |
|------|------|
| **유지보수성** | 접속 정보를 코드와 분리해 외부에서 관리 |
| **유연성·확장성** | 리소스를 중앙에서 집중 관리 |
| **자원 공유** | 서로 다른 애플리케이션 간 자원 공유 가능 |

## 📝 정리

```
JNDI
├─ 개념      이름(name) ↔ 객체(object) 매핑, 이름으로 리소스 접근
├─ 서버 설정 server.xml <Resource ...> 로 DataSource 등록
├─ 앱 사용   InitialContext.lookup("java:comp/env/...")
└─ 장점      접속 정보 분리 → 유지보수성·유연성·자원 공유
```

| 개념 | 한 줄 정의 |
|------|------|
| **JNDI** | 이름으로 리소스에 접근하는 자바 API |
| **Resource(server.xml)** | 컨테이너가 관리하는 리소스 등록 |
| **validationQuery** | 커넥션 유효성 검사 쿼리 (DB별로 다름!) |

JNDI는 "접속 정보를 코드가 아니라 서버가 관리한다"는 발상이다. 특히 `validationQuery`는 DB 종류에 맞게 작성해야 한다는 점을 기억하자 — 사소해 보이지만 실무에서 발목을 잡는다.
